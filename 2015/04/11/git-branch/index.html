在来到新公司之后，使用Git作为源代码版本控制管理的工具，虽然一直在用git来管理自己的代码，但是还一直没有使用过分支管理。在以前的工作中，也是用过svn的分支管理，但是那种分支管理使用起来非常不方便，新建分支和切换分支都非常不方便。在svn里基本很少人愿意使用分支管理。但是Git不一样，Git可以很方便的来管理分支。所以在Git中分支管理是一个非常常用的一个功能。

为什么要使用分支管理

在一个项目开发周期，假设有一个任务，你需要两周完成，但是你在第一周的时候提交了你的50%的代码，因为代码还没写完，导致当前代码库里的代码是编译不能通过的状态，这时就影响了别人的开发。或者其他已开发好的模块希望尽快上线，这时也不能上线了。如果自己建立一个分支，自己编写的代码只提交到自己的分支，和主分支分离，这样就不会相互影响了。

分支管理最佳实践

上面主要介绍了git分支管理的命令，下面我们来介绍下git分支管理的最佳实践：
git有两个主要的分支：master,develop。
master：管理要上线的代码。主要用于生产上线。
develop：来管理开发代码。主要用于开发集成测试。
下面我们分两个场景：
1、新的开发任务
新的开发任务我们先从master或者develop建立一个分支，可以命名为：日期名字，或者日期任务Id_名字。作为自己分支来作为开发。开发完成后，需要合并至develop分支，进行集成测试，集成测试完成后，就可以合并至master分支用于生产上线。
2、Bug修复
在线上运行期间，需要修复紧急bug，可以从master建立一个分支，然后修复紧急bug,，并需要合并至master分支以及develop分支。如果是测试MM测试过程中发现bug，如果可以在下个版本修复的话，可以先在自己当前分支修复，最后会合并至master，如果比较紧急，可以使用线上运行期间发现bug的处理方式建立分支。
